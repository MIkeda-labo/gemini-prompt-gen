# Gemini Prompt Generator 技術・仕様書

## 1. プロジェクト概要
本プロジェクトは、ユーザーが入力した「目的」に対して最適なプロンプトを構築し、さらに「除外すべき要件（Not変数）」や「ローカルのドキュメント（RAG）」を加味した上で、Gemini APIを用いて最終的な高度な出力を得るためのAIエージェントアプリケーションです。

## 2. 技術選定と採用理由

### 2.1 Next.js (App Router) / React
*   **一般的説明**: ReactをベースにしたフルスタックWebフレームワーク。App Routerはサーバーコンポーネントとクライアントコンポーネントを明確に分離した最新のルーティング手法です。
*   **選定理由**: PWAライクなリッチなUI（Glassmorphism等）の構築が容易である点と、後述するServer Actionsを利用したバックエンド処理の統合がシームレスに行えるため採用しました。

### 2.2 React Server Actions
*   **一般的説明**: Next.jsに組み込まれた機能で、クライアント側（ブラウザ）から直接サーバー側の関数を呼び出せる仕組みです。
*   **選定理由**: 従来の `/api/xxx` のようなAPIエンドポイントを個別に作成する手間を省きつつ、**APIキー（GeminiやOpenAI）をブラウザ側に一切露出させない（セキュアに保つ）**という極めて重要なセキュリティ要件を満たすためです。

### 2.3 Gemini API (`@google/genai` SDK) & Code Execution機能
*   **一般的説明**: Googleが提供する高性能な大規模言語モデル（LLM）API。特に `Code Execution` ツールは、プロンプトの指示に従ってAI自身がクラウド上のサンドボックス内でPythonコードを書き、その実行結果をレスポンスに含めることができる先進的な機能です。
*   **選定理由**: 「プロンプトの最適化」や「除外理由の下書き生成」などの中核的な推論を担います。また、ローカルでPython環境（`numpy`等）を構築する手間を省くため、Code Execution機能を利用して「Not変数に似た概念の抽出」をクラウドで完結させる構成としました。

### 2.4 OpenAI Embeddings API (`text-embedding-3-small`)
*   **一般的説明**: テキストを多次元の数値配列（ベクトル）に変換するAPIです。
*   **選定理由**: 文章の意味的な近さを計算する（RAG）ために必要不可欠です。安価で精度が高く、ブラウザ側でのコサイン類似度計算と組み合わせて高速な文書検索を実現するために採用しました。

### 2.5 File System Access API (`showDirectoryPicker`)
*   **一般的説明**: モダンブラウザに搭載されている、ユーザーのローカルファイルやディレクトリをブラウザから直接読み書きするためのJavaScript APIです。
*   **選定理由**: ユーザーのローカルにある秘匿性の高いドキュメント（.mdや.txtファイル）をわざわざサーバーにアップロードすることなく、**ブラウザのメモリ上だけで解析・検索（インフローRAG）**できるようにするためです。

## 3. 主要なディレクトリ・ファイル構成

*   `app/page.jsx`
    *   **役割**: クライアントUI（Client Component）、状態管理（Hooks）、ローカルファイルの読み込みとチャンク化、およびコサイン類似度計算によるRAG検索の実行。
*   `app/actions.js`

*   `.env.local`
    *   **役割**: 秘匿情報（`GEMINI_API_KEY`, `OPENAI_API_KEY`）を管理。

## 4. 機能ごとのワークフローと処理ロジック（最小コード例）

### 4.1 プロンプト生成フロー
ユーザーが入力した「目的」から、重視したい観点（Perspective Tags）を選び、最適化されたプロンプトを作成します。この時点では「禁止事項（Not変数）」や「ローカル検索（RAG）」は関与させず、純粋な指示書の精度を高めることに特化します。

*   **関連ファイル**: `app/page.jsx`, `app/actions.js`

```javascript
// app/actions.js: プロンプト生成 (抜粋)
export async function generateOptimizedAct(intent, currentPrompt, tagsStr) {
  const response = await ai.models.generateContent({
    model: 'gemini-2.5-flash',
    contents: `目的: ${intent} ... (省略) ...`
  });
  return JSON.parse(response.text);
}
```

### 4.2 Code Executionを利用した「除外理由（下書き）」の動的生成
「失敗しやすい・除外すべき項目」が選択された際、ローカル環境のPythonには頼らず、Geminiの**Code Execution**を利用して「類似する禁止アプローチ」を自動推論させます。

*   **関連ファイル**: `app/actions.js`

```javascript
// app/actions.js: Code Executionによる推論 (抜粋)
export async function generateRAGReasoningAct(notParamsStr) {
  // ① Pythonをクラウド実行し、類似概念を算出
  const pythonResponse = await ai.models.generateContent({
    model: 'gemini-2.5-flash',
    contents: `以下のキーワードに対して、Pythonコードを実行して意味的に近い概念を推測しなさい。キーワード: ${notParamsStr}`,
    config: { tools: [{ codeExecution: {} }] } // ← ここでPythonサンドボックスを利用
  });
  
  const pyResultText = pythonResponse.text;
  
  // ② 算出結果を含めて、除外すべきルール（Negative Prompt）の下書きを文章化
  const draftResponse = await ai.models.generateContent({
    contents: `以下の内容を除外すべき理由を簡潔に文章化しなさい。\n${pyResultText}`
  });
  return draftResponse.text;
}
```

### 4.3 ローカルRAGを利用した「最終実行（Gemini）」フロー
ユーザーが「下書き」を編集・保存し、ローカルディレクトリをブラウザへ読み込ませた後、最終的なアンサーを生成します。

*   **関連ファイル**: `app/page.jsx`

```javascript
// app/page.jsx: RAG抽出と実行 (抜粋)
async function executeWithGemini() {
  let ragTexts = [];
  
  // ① 読み込まれたローカルファイル(localDocs)がある場合、ベクトル検索を実行
  if (localDocs.length > 0) {
    // 生成済みの最適化プロンプトを検索クエリとしてベクトル化
    const queryEmbeds = await getOpenAIEmbeddingsAct([output.optimized]);
    const queryVec = queryEmbeds[0];
    
    // JS上でコサイン類似度を比較し、関連性の高い上位チャンクを抽出
    const scoredDocs = localDocs.map(doc => ({
      ...doc,
      score: cosineSimilarity(queryVec, doc.embedding)
    })).sort((a, b) => b.score - a.score);
    
    ragTexts = scoredDocs.slice(0, 5).map(d => d.text);
  }

  // ② 抽出したRAG知識(ragTexts)と、手動で修正した禁止条件(ragReasoning)をセットで最終実行
  const res = await executePromptAct(output.optimized, ragReasoning, ragTexts);
  setExecutionResult(res);
}
```

## 5. 本設計の特筆すべきポイントと結論
*   **段階的処理の分離**: 「プロンプトの構成」と「知識ベース（RAG）の適用・禁止事項の適用」をフェーズとして分けることで、過度なプロンプトインジェクションやハルシネーションを防ぐ非常に保守性の高いシステムを実現しました。
*   **サーバーレス志向のアーキテクチャ**: ローカルの複雑なPython環境（`child_process` 等）を排除し、Next.js Server Actions と API(Gemini Code Execution) のみで完結させることで、プラットフォームに依存しないポータビリティの高いアプリケーションとなっています。

## 6. 今後の課題と制約事項（Next Steps）
アーキテクチャの性質上、本格的に運用を拡大する際に発生しうる課題と解決に向けた方針です。

### 6.1 RAG用ファイルの大量読み込み時のクラッシュ限界
*   **課題**: 現在の「ブラウザで読み込む → メモリ上に保持する → OpenAI Embeddings APIに一括送信する」というフローは、対象となるファイル数やテキスト量が膨大（数千ファイル等）になると、**ブラウザのメモリ不足（タブのクラッシュ）やOpenAIやGemini APIのペイロード上限エラー**を引き起こします（※現在は応急処置としてRAGの抽出チャンク数を絞っています）。
*   **解決案**: ベクトルデータベース（Pinecone等）を導入し、読み込んだ情報を恒久的に保存・検索できるバックエンド構造へ移行する必要があります。

### 6.2 File System Access APIの制限（フォルダの毎回選択問題）
*   **課題**: ブラウザのセキュリティ仕様上、リロードのたびにローカルフォルダへのアクセス権限が消滅するため、毎回「読み取りボタン」からユーザー自身でフォルダを選択し直す必要があります。完全な自動同期はブラウザベースでは不可能です。
*   **解決案**: アプリを Electron や Tauri などの「デスクトップアプリケーション」としてパッケージ化することで、ローカルファイルへの無制限・永続的アクセス権を取得できます。

### 6.3 類似度検索の精度と「トークンの希釈」
*   **課題**: 単純なコサイン類似度（Vector Search）の上位チャンクをそのままGeminiに渡していますが、「文脈の欠落」や「ノイズの混入」が起きる可能性があります。
*   **解決案**: 取得後に一度別の軽量モデルでフィルタリングや要約を行う「Advanced RAG（Re-ranking等）」のパイプライン導入が効果的です。
